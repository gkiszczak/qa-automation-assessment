# Performance Test Report (k6) — reqres.in

## Target
- Endpoint: `GET https://reqres.in/api/users?page=1`

## Goal (from assignment)
- 100 concurrent users
- each user sends ~1 request / second
- capture: P50/P95/P99, error rate, throughput

## Tool
- k6 (run via Docker; no local k6 installation required)

## How to run
Prerequisite: Docker.

```bash
npm run perf
```

CI-style (also generates artifacts):
```bash
npm run perf:ci
```

Outputs (generated by k6 `handleSummary`):
- `performance/output/summary.json`
- `performance/output/performance-report.generated.md`

## Test setup & assumptions
Defaults (can be overridden via env vars):
- **VUs**: 100 (`K6_VUS`)
- **Target rate**: 100 iterations/sec (`K6_RPS`)
  This approximates **100 users × 1 req/sec** => **~100 RPS total**.
- **Duration**: 60s (`K6_DURATION`)
  Chosen to allow warm-up and capture stable percentiles.

> Note: `reqres.in` is a public demo API; results may vary due to shared infrastructure or rate limiting.

## Interpretation guidance
- Large gap between P50 and P95/P99 typically indicates tail latency (jitter, saturation, slow paths).
- If error rate increases under load, likely rate limiting or upstream capacity constraints.

## Potential optimization suggestions (generic)
- Enable caching with ETag/Cache-Control where applicable.
- Compress JSON responses (gzip/br).
- Ensure DB queries are indexed; avoid N+1 patterns.
- Use circuit breakers / bulkheads for downstream dependencies.
- Consider CDN/edge caching for global traffic.
