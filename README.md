# QA Automation & Performance Assessment

This repository contains:
- UI automation tests for https://www.saucedemo.com using **Playwright Test**
- A performance test for https://reqres.in using **k6**
- Short seniority reflection answers

## Prerequisites
- Node.js 18+ (recommended 20)
- npm
- (optional) k6 for performance testing

## Install
```bash
npm install
npx playwright install
```

## Run UI tests locally
```bash
npm test
```

### Open HTML report
```bash
npm run report
```

### Run with UI mode (debug)
```bash
npm run test:ui
```

## Project structure
- `pages/` — Page Objects (encapsulate selectors and business actions)
- `fixtures/` — Playwright fixtures for reusable setup (login)
- `tests/` — test specs (readable, deterministic, no hard sleeps)
- `performance/` — k6 script + performance report

## Design decisions (why it is structured this way)
- **Page Objects** keep selectors and UI details in one place.
- **Fixtures** keep login reusable and avoid copy-paste between tests.
- **Determinism**: no `waitForTimeout`; expectations and Playwright auto-waits are used.
- **Stability**: selectors rely on IDs / data-test where possible; item selection is done by item name + container scoping.

## Task 1 — Covered scenarios
- Successful login: `standard_user`
- Failed login: `locked_out_user`
- Add a product to cart
- Start checkout process (reach `checkout-step-one.html`)
### **Additional design idea**: adding to cart and starting checkout scenarios could be separated. It would be more efficient to use API to add item to cart and start checkout scenario.

## Task 2 — Performance testing (k6 via Docker)

No local k6 installation is required — tests run using the official `grafana/k6` Docker image.

### Run performance test
Prerequisite: Docker.

```bash
npm run perf
```

### CI-style run (also generates artifacts)
```bash
npm run perf:ci
```

Artifacts generated by k6 (via `handleSummary`):
- `performance/output/summary.json`
- `performance/output/performance-report.generated.md`

Load can be tuned using env vars:
- `K6_VUS` (default 100)
- `K6_RPS` (default 100)
- `K6_DURATION` (default 60s)

# Reflection & Seniority Check

## How would you integrate Playwright tests into CI/CD?
- Run UI tests on every Pull Request and on merges to main.
- Keep tests fast and reliable: tag and separate slow tests; run critical smoke tests first.
- Publish artifacts (HTML report, traces) and make failures actionable.
- Use retries only as a safety net (e.g. 1 retry in CI) and track flakiness separately.

## How would you notify the team (Slack/Teams) about failures or regressions?
- Slack/Teams webhook from CI (GitHub Actions) on failure with:
  - link to run
  - failing spec(s)
  - link to HTML report artifact
- For performance regressions: alert on threshold breaches (P95 / error rate) via Grafana/Datadog.

## What observability metrics would you include in an end-to-end quality dashboard?
- CI: pass rate, flaky rate, mean time to repair, build duration, test duration
- Product: error rate, latency (P50/P95/P99), throughput, availability, SLO compliance
- User journey: success rate for key flows (login/checkout), client-side errors (JS), API errors
- Release: deployment frequency, change failure rate, rollback rate

## How would you decide what to automate / what not to automate?
Automate when:
- High business value / high risk areas (critical paths)
- Stable behavior and clear assertions
- Repeated regression checks (cheap to run, catches common issues)

Do not automate when:
- Requirements/UI are volatile and will cause high maintenance cost
- Assertions are subjective (exploratory, UX polish)
- One-off scenarios with low ROI

## What belongs to performance vs functional testing?
- **Functional**: correctness for given inputs (workflow works, correct validation, expected UI/API behavior).
- **Performance**: correctness *under load* and non-functional qualities (latency percentiles, throughput, error rate, resource saturation).

## Performance Testing Notes

During the implementation of performance tests several issues were identified.


### Authentication
Initial requests failed with **100% error rate** because the API requires an authentication header:

```
x-api-key
```

The header was added and the key is provided via the environment variable:

```
REQRES_API_KEY
```

to avoid hardcoding credentials.

---

### Cloudflare Protection
Requests executed from GitHub Actions sometimes return:

```
403 Forbidden
```

with a Cloudflare challenge page (`Just a moment...`).  
This indicates the public demo API (`reqres.in`) blocks automated traffic from cloud infrastructure.

Because k6 is a headless HTTP client, it cannot solve JavaScript-based Cloudflare challenges.

---

### Demo API Limitations
`reqres.in` is a public demo API and not intended for load testing.  
Observed limitations include:

- rate limiting
- WAF protection
- inconsistent responses from CI environments

These factors may affect performance test stability.

---

### Improvements
The implementation includes:

- custom metrics (`status_2xx`, `status_4xx`, `status_5xx`, `status_429`)
- limited error logging for debugging
- CI artifacts:

```
performance/output/summary.json
performance/output/performance-report.generated.md
```
